# 产品业务基线档案：AllMall AI Commerce Ops

> 审计标准：PM-Audit-Skill v3 | 基线版本：2026-07-10-v5 | 范围：全产品体系

---

## 一、产品定位与核心价值

**AllMall** 是面向电商卖家的 AI 自动化运营 SaaS 产品，定位为「AI Commerce Ops」。通过 15 个 AI Agent 覆盖电商运营全链路，替代人工执行重复性运营任务。

**产品核心价值**：
- 降本：AI 替代人工执行定价、广告投放、客服回复、评价管理等重复运营
- 增效：Agent 7×24 自动运行，响应速度远超人工
- 增收：智能定价策略、广告优化、库存预警帮助卖家提升 GMV

**当前产品阶段**：内测/原型演示，全前端 Mock 数据，无真实后端。

## 二、目标用户与用户分层

### 2.1 目标用户

| 用户群体 | 典型画像 | 核心痛点 | 使用频次 |
|---------|---------|---------|:--:|
| 中小跨境卖家 | 经营 1-5 家 TikTok/Amazon 店铺，团队 1-3 人 | 人力有限、多平台管理复杂、运营知识不足 | 日活 |
| 品牌电商团队 | 自有品牌，5-20 家店铺，专职运营团队 | 团队协作、运营标准化、数据驱动决策 | 日活 |
| 代运营机构 | 代管多个客户店铺，规模化管理 | 多客户管理、效率规模化、客户汇报 | 日活 |

### 2.2 用户角色与权限体系

| 角色 | 职责 | 权限范围 |
|------|------|---------|
| **Owner** | 店铺所有者，全局控制 | 全权限（查看/配置/操作/审批/成员管理/订阅管理） |
| **Admin** | 管理员，团队管理 | 全权限（同 Owner，不可删除团队） |
| **Operator** | 运营人员，日常操作 | 可执行任务、管理商品/订单、查看经营数据 |
| **Approver** | 审批人 | 仅可审核/驳回审批，查看关联任务 |
| **Finance** | 财务 | 仅查看财务台账，管理订阅 |
| **Viewer** | 观察者 | 全量只读，不可操作 |

> **当前状态（代码实现）**：domain.ts 第 115 行 Member.role 类型仅定义了 5 个角色（Owner/Admin/Operator/Approver/Viewer），**Finance 角色尚未在代码中实现**。i18n 翻译键 mvpRoles 展示 5 个角色（含 Viewer）。成员邀请 Modal 的 roleOptions 仅提供 4 个可选角色（Admin/Operator/Approver/Viewer，不含 Owner）。角色权限拦截尚未在任何页面执行。

### 3.2 Agent 五层架构（共 15 个 Agent）

| 层级 | Agent | 核心业务价值 |
|------|-------|------------|
| **基础管道层** | 店铺保活、市场情报、商品上架、素材工厂 | 保障店铺基础运行，自动化商品上下架流程 |
| **流量引擎层** | 广告投放、定价策略、**直播运营** | 驱动流量获取与转化，智能优化投放 ROI |
| **增值运营层** | CRM 留存、评价管理、客服代理、售后处理、**促销管理** | 提升客户体验与复购率 |
| **支撑服务层** | 库存预警、财务对账 | 供应链与资金流健康保障 |
| **独立系统层** | 风险控制 | 全链路安全护栏 |

> **注意**：product-design.md 的 Agent 体系全貌表格仅收录了 13 个 Agent，**遗漏了 `live_stream_ops`（直播运营）和 `promotion_campaign`（促销管理）两个 Agent 的详细设计**。代码实现中 15 个 Agent 均已完整实现（agentMockData.ts、i18n.tsx、AgentListPage/BuiltinTasksSection 均有覆盖），属于产品设计文档滞后于代码实现。

## 四、核心业务场景

### 4.1 卖家开通自动运营场景
1. 卖家登录 → 进入经营总览 → 看到「尚未连接店铺」提示 → 点击「快速配置」
2. 快速配置页：① 选择运营场景，AI 自动开通对应 Agent → ② 选择/新增店铺 → ③ 配置定价/广告/客服参数 → ④ 保存 → Agent 开始运行
3. 卖家返回 Dashboard，可见 Agent 成果卡片展示 AI 运行效果

### 4.2 日常运营监控场景
1. 卖家每日打开 Dashboard → 查看 Agent 今日成果（节省时间/增收/处理事务）→ 查看异常提醒（差评待回、库存预警、ROI 偏低）
2. 点击具体异常 → 跳转到对应处理页面
3. 在 Agent 列表中查看各 Agent 运行状态和成功率

### 4.3 高风险操作审批场景
1. Agent 自动创建高风险任务（如大批量调价、退款）→ 生成审批
2. 审批中心显示待审批项 → Approver 角色审核
3. 审批通过 → Agent 继续执行；驳回 → 任务取消
4. 审批详情页展示完整操作时间线

### 4.4 演示体验场景
1. 未登录用户点击「体验演示」→ 进入演示模式
2. 顶部黄色横幅提示「演示模式 · 数据为模拟数据」
3. 可浏览全部页面，体验完整运营闭环
4. 点击「退出演示」→ 返回登录页

### 4.5 试用转化场景
1. Free 套餐用户 Dashboard 顶部显示试用横幅
2. 提示剩余试用天数、已用店铺数、Agent 调用量
3. 点击「升级套餐」→ 跳转财务台账 → 选择 Professional/Enterprise 套餐

## 五、产品规则体系

### 5.1 订阅规则
- Free 套餐：1 店铺、每月 500 Agent 调用、3 个基础 Agent
- Starter 套餐：3 店铺、每月 2000 Agent 调用、9 个 Agent
- Professional 套餐：10 店铺、每月 10000 Agent 调用、15 个 Agent
- Enterprise 套餐：无限制店铺、自定义 Agent 调用量、全部 Agent + 专属支持

### 5.2 Agent 运行规则
- 基础管道层 Agent 必须按序启用：保活 → 情报 → 上架 → 素材
- 启用上层 Agent 时必须已启用其所有依赖 Agent
- 风险控制为全链路护栏，启用后包裹所有 Agent 操作
- login_bootstrap（店铺保活）是系统必须 Agent，不可禁用

### 5.3 审批规则
- 退款、大批量调价、高风险配置变更需经审批
- 审批按风险等级分级：高风险 → Approver 角色审批
- 审批驳回后任务回到初始状态，运营可重新提交

### 5.4 数据口径
- GMV：已完成订单总金额（不含退款）
- 订单：所有状态的订单计数
- Agent 成功率：`successTasks / totalTasks × 100%`
- 试用天数：从首次登录起 14 天

## 六、用户操作路径

### 6.1 主要用户旅程

```
登录 → Dashboard（查看成果/异常）
  ├─ 有店铺有 Agent → 日常监控 → 处理异常/审批
  ├─ 有店铺无 Agent → 前往 Setup → 开通 Agent → 配置参数 → 返回 Dashboard
  ├─ 无店铺 → 前往 Setup 或 体验演示
  └─ 未登录 → 登录页 → 体验演示 → 浏览 → 退出或登录
```

### 6.2 关键操作路径

| 起点 | 目标操作 | 完整路径 |
|------|---------|---------|
| Dashboard | 处理库存预警 | Dashboard 提醒 → 点击 → 商品管理页 → 调整库存 |
| Dashboard | 处理差评 | Dashboard 提醒 → 点击 → Agent 评价管理 → 回复差评 |
| Agent 列表 | 查看 Agent 详情 | 点击「配置」→ Agent 配置页 → 查看策略/任务/日志 |
| Agent 列表 | 调整策略 | 点击「配置」→ 策略配置区域 → 修改参数 → 自动保存 |
| 异常中心 | 处理异常 | 筛选 → 点击异常 → 分配/处理/关闭 |
| 审批中心 | 审批操作 | 筛选 → 点击审批 → 查看详情 → 通过/驳回 |

## 七、既定产品约束

1. 全前端 Mock 数据，无真实后端连接
2. 所有数据存储在内存变量中，刷新页面后恢复初始值
3. 登录认证为硬编码演示账号（admin@allmall.com / demo123）
4. 产品支持中英文双语，基于 Ant Design ConfigProvider + 自定义 i18n
5. Demo 模式通过 React Context 控制，状态基于 sessionStorage 持久化
6. 所有 Agent 配置修改使用 React Query 的 queryClient.setQueryData 不可变更新
7. 演示原型定位：电商运营全链路 AI 自动化能力展示
