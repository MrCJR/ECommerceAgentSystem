AllMall Database Design · V0.1

SaaS 平台数据库设计

本文档定义 AllMall 第一阶段核心数据模型，用于支持租户隔离、店铺授权、任务编排、审批、审计、额度和 MuleRun session 绑定。

1. 设计原则

  • 所有业务数据必须带 tenant_id，涉及店铺的数据必须带 store_id。
  • 核心表使用 UUID / ULID 作为主键，方便跨服务和事件系统引用。
  • 状态字段使用枚举或受控字符串，并在应用层维护状态机。
  • 敏感密钥不直接明文落库，Secret Vault 只在业务表中保存引用。
  • 审计日志不可随意修改，优先采用 append-only 思路。

2. 核心实体总览

领域核心表说明
租户与权限tenants, users, roles, user_roles公司、成员、角色和权限关系。
店铺授权stores, store_authorizations, connect_tokens平台店铺、授权状态、MuleRun session 绑定。
任务控制面tasks, workflows, runs, steps业务目标、流程模板、执行实例和步骤。
审批approvals, approval_actions高风险动作审批与审批历史。
审计与证据audit_logs, artifacts, agent_events执行日志、截图、trace、事件回调。
额度与计费plans, subscriptions, quota_usages, billing_records套餐、订阅、资源消耗和账单流水。

3. 店铺授权模型

stores
- id
- tenant_id
- platform
- store_name
- external_store_id
- status
- created_at
- updated_at

store_authorizations
- id
- tenant_id
- store_id
- auth_mode                 -- mulerun_browser_profile / oauth / manual_secret
- runtime_provider          -- mulerun / claw_runner / platform_api
- runtime_session_id
- secret_ref
- status                    -- pending_login / connected / login_required / revoked / expired
- last_verified_at
- revoked_at
- created_at
- updated_at

connect_tokens
- id
- tenant_id
- store_id
- token_hash
- purpose                   -- login_bootstrap / re_auth
- status                    -- active / used / expired / revoked
- expires_at
- used_at
- created_by
- created_at
AllMall 保存的是 runtime_session_id 这种引用，不直接把 MuleRun 沙盒中的长期 cookie 当成普通业务字段保存。

6. 后续待定问题

  • 企业版是否采用独立数据库 / 独立 schema，标准版是否采用共享库 + tenant_id。
  • 是否在 PostgreSQL 中启用 Row Level Security。
  • Secret Vault 选型：云厂商 KMS、HashiCorp Vault，或阶段性自建加密存储。
  • 审计日志是否需要进入独立存储或冷归档。
