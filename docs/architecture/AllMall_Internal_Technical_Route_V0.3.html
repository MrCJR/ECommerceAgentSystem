AllMall Internal Route · V0.3

双团队内部协作路线KickOff

本文档用于启动 SaaS 平台团队与 Agent 团队的并行研发。它不是 API 细节文档，也不是数据库设计文档，而是定义两支团队在内测阶段如何分工、如何联调、如何验收，以及如何从 MuleRun 验证期演进到标准 Agent Runtime。

启动会主文档 SaaS Control Plane MuleRun Agent Lab 国内电商平台验证

1. 内测目标

内测阶段的目标不是一次性完成最终系统，而是让两个团队尽早开始独立研发，并通过一个最小但真实的业务闭环完成验证。

1

完成店铺绑定

通过 connectToken 与 MuleRun browser session 建立绑定，证明真实店铺登录和会话复用可行。

2

跑通首个 Agent

首个高价值业务 Agent 确定为竞品调研 Agent，以拼多多店铺和同类商品为第一验证场景。

3

回传执行证据

Agent 执行过程需要回传状态、步骤、截图、日志、结果摘要和失败原因。

4

建立协作节奏

每周通过演示、问题清单、接口规范和验收标准推动两队同步前进。

启动会目标：SaaS 团队负责控制台和业务主控，Agent 团队负责在 MuleRun 中证明真实执行能力，双方共同跑通“店铺绑定 -> Agent 执行 -> 事件回传 -> 审批/审计展示”的闭环。

2. 协作原则

AllMall 是控制面

租户、店铺、任务、审批、审计、额度和状态流转由 AllMall 管理。MuleRun 不承担业务系统主控权。

MuleRun 是执行实验室

内测阶段 MuleRun 承担浏览器沙盒、Agent 调试、session 复用和执行过程验证。

人机校验由人完成

短信验证码、扫码、滑块验证和平台风控校验不设计为自动绕过，而是通过 Human Session Bootstrap 完成合法登录。

接口规范小而稳定

主链路只围绕 connectToken、runtimeSessionId、task/run、event callback、artifact 和 approval 展开。

内测阶段由 MuleRun 保存 browser profile / session。AllMall 不把长期店铺密码、长期 cookie 或完整 token 作为普通任务参数传递。AllMall 保存 session reference，并保留吊销、重绑和审计能力。

4. 团队职责边界

团队 主要职责 启动会后第一批交付物 不负责什么
SaaS 平台团队 负责 arkops-portal、后端控制面、店铺授权、任务中心、审批、审计、artifact 展示和联调 API。 店铺列表与详情、connectToken 生成、任务创建、执行时间线、事件回调接收、审批列表、截图日志展示。 不负责在 MuleRun 内部调试 Agent，不直接处理店铺长期 cookie，不直接调用电商平台页面。
Agent 团队 负责在 MuleRun Web 平台调试 Login Bootstrap Agent 与首批高价值业务 Agent。 登录绑定 SOP、session 复用验证报告、首个 Agent SOP、事件回传样例、截图日志样例、失败场景清单。 不负责 SaaS 租户系统、权限系统、计费系统和 Portal 页面开发。
共同负责 负责联调规范、状态机、事件类型、验收标准和问题升级。 每周演示记录、接口变更记录、联调问题清单、MVP 验收结果。 不在口头约定中修改核心接口；接口变更必须同步到 API 文档。

5. Agent 团队工作规划

Agent 团队第一阶段不搭建完整平台，核心任务是在 MuleRun 中证明真实执行能力。工作拆分为两条并行主线。

Workstream A

Session Bootstrap 线

  • 设计拼多多店铺 Login Bootstrap Agent。
  • 验证用户手动输入 connectToken 的流程。
  • 验证浏览器沙盒 / profile 是否可持久化。
  • 验证二次任务能否复用已登录 session。
  • 验证 session 失效、人机校验再次出现时，是否能触发 re-auth。
Workstream B

高价值业务 Agent 线

  • 首个业务 Agent 确定为竞品调研 Agent。
  • 定义 Agent 输入：店铺、商品类目、目标商品、竞品范围、时间范围、调研维度。
  • 定义 Agent 输出：竞品清单、价格区间、活动信息、销量/评价线索、内容素材差异、机会点和证据截图。
  • 识别需要人工确认的数据口径、竞品选择和结论风险。
  • 沉淀可复用 SOP，区分“数据采集”“分析归因”“人工确认”“报告生成”。
Agent 团队启动重点：Agent 团队无需等待 SaaS 平台完全完成。第一阶段在 MuleRun 中跑通真实登录、真实页面读取、真实截图日志和真实结果摘要；SaaS 团队通过 mock API 和最小回调入口逐步接入。

6. Agent 团队交付物标准

Agent 团队的交付物不是一次性 MuleRun demo，而是一套可被 SaaS 平台接管、可复用、可验收、可沉淀为产品能力的 Agent 工作包。每个进入联调的业务 Agent 都必须按照统一结构交付。

标准交付定义：Agent SOP + MuleRun 可运行原型 + 输入输出规范 + 执行证据包 + 异常与人工确认清单 + 联调验收报告。

6.2 交付物一：Agent SOP 文档

SOP 文档描述 Agent 如何完成真实运营任务，是业务团队、Agent 团队和 SaaS 团队共同理解 Agent 行为的基础文件。

  • 业务目标：说明 Agent 解决什么问题。竞品调研 Agent 的目标是帮助运营人员调研拼多多同类商品的价格、活动、销量/评价线索、标题、主图、详情页卖点和机会点。
  • 使用前提：店铺已在 AllMall 中创建，MuleRun session 可用，用户提供目标商品、商品类目或关键词。
  • 执行步骤：进入平台、搜索关键词、筛选竞品、采集商品信息、截图留证、汇总对比、生成报告。
  • 人工确认点：竞品选择、数据口径、结论入报告范围和对外同步范围。
  • 禁止动作：不自动改价、不自动报名活动、不自动修改商品、不自动下架商品、不自动联系客户。
  • 成功标准：输出完整竞品清单、关键截图、数据来源、对比摘要和机会点。
  • 失败处理：登录失效、页面加载失败、平台风控、搜索结果异常、数据不完整时，输出明确事件和失败原因。

6.3 交付物二：MuleRun Agent 原型

MuleRun Agent 原型是在 MuleRun 平台中可运行、可复用、可联调的 Agent 配置和执行流程。该原型不是一次性演示，而是后续 SaaS 接管和产品化验证的执行模板。

  • MuleRun workflow / Agent 配置说明。
  • Agent prompt / instructions / tool calling 流程。
  • 浏览器 profile 或 session 使用方式。
  • connectToken 输入与店铺 session 绑定流程。
  • 竞品调研任务启动方式。
  • 每一步如何截图、记录日志、返回事件。
  • 当前版本限制、已知问题和不可自动化部分。

6.4 交付物三：输入输出协议样例

输入输出协议决定 SaaS 平台能否稳定创建任务、展示结果和沉淀数据。该协议后续进入 API 文档，成为 SaaS 团队与 Agent 团队的正式联调规范。

输入字段

  • storeId：ArkOps 店铺标识。
  • runtimeSessionId：MuleRun session 引用。
  • agentType：Agent 类型，第一阶段为 competitor_research。
  • 商品类目、目标商品链接或关键词。
  • 竞品范围、调研维度、时间范围。
  • 是否需要人工确认。

输出字段

  • 竞品商品列表、商品标题、价格区间。
  • 活动信息、销量/评价线索、内容素材差异。
  • Agent 分析结论和机会点。
  • 截图、日志和原始数据引用。
  • 是否需要人工确认，以及需要确认的原因。

6.5 交付物四：执行证据包

Agent 每次运行必须留下可审计证据。证据包进入 AllMall 任务时间线、审计日志和后续模型训练数据。

  • 每个关键步骤的截图。
  • 页面 URL 或页面标识。
  • 采集到的原始数据。
  • Agent 的中间判断与最终报告。
  • 错误日志和失败原因。
  • 执行开始时间、结束时间和耗时。
  • taskId、runId、stepId 关联信息。

6.6 交付物五：异常与人工确认清单

Agent 团队必须把真实执行中遇到的问题沉淀为产品状态、事件类型和人工确认机制，而不是停留在个人调试经验中。

异常类型 表现 平台侧转化
登录失效 拼多多要求重新登录、扫码、短信验证码或滑块验证。 login_required，进入重新授权流程。
页面结构变化 按钮、字段、价格位置、活动标签或商品卡片结构变化。 step_failed，记录页面截图和失败步骤。
数据口径不确定 存在券后价、拼单价、活动价、会员价等多个价格口径。 manual_review_required，请求人工确认数据口径。
竞品选择不确定 搜索结果中存在非同类商品、广告商品或品牌不相关商品。 approval_required，请求人工确认竞品清单。
结论置信度不足 数据缺失、样本不足或多个信号互相冲突。 data_uncertain，报告中标记低置信度结论。

6.7 交付物六：联调验收报告

每个 Agent 进入联调后必须形成验收报告，证明它具备可重复运行、可追踪、可接管的产品化基础。

  • 测试平台：第一阶段为拼多多。
  • 测试店铺数量、测试任务数量、成功次数和失败次数。
  • 平均执行时间和最长执行时间。
  • session 是否复用成功。
  • 是否触发过人工确认或重新登录。
  • 是否能回传截图、日志、原始数据和结果摘要。
  • SaaS 平台是否能展示完整任务时间线。
  • 当前不能自动化的部分和下一阶段优化事项。

7. SaaS 团队工作规划

店铺授权闭环

  • 创建店铺
  • 生成 connectToken
  • 展示 pending_login / connected / login_required / revoked
  • 接收 MuleRun session binding 回调

任务执行闭环

  • 创建 Task
  • 展示 Run / Step 时间线
  • 接收 Agent events
  • 展示 screenshots / logs / summary

审批审计闭环

  • 处理 approval_required
  • 记录审批人、理由和结果
  • 所有回调进入 Audit Log
  • 为后续模型训练保留证据链

8. 第一阶段联调场景

启动会后冻结一个主场景，避免两个团队同时追多个平台和多个 Agent。第一阶段联调场景确定如下：

基础设施场景：拼多多店铺登录绑定

  1. 用户在 AllMall 添加拼多多店铺。
  2. AllMall 生成短期 connectToken。
  3. 用户打开 MuleRun Login Bootstrap Agent 并输入 connectToken。
  4. 用户在 MuleRun 浏览器沙盒中手动登录并完成校验。
  5. MuleRun 回传 runtimeSessionId。
  6. AllMall 将店铺状态更新为 connected。

业务场景：竞品调研 Agent

  1. AllMall 创建竞品调研任务。
  2. MuleRun 加载对应店铺 session。
  3. Agent 读取目标商品、竞品商品、价格、活动、评价和内容素材信息。
  4. Agent 回传步骤、截图、数据来源、竞品对比结论和报告摘要。
  5. 涉及竞品选择、数据口径或结论发布时，触发人工确认。
  6. AllMall 展示时间线、人工确认记录和审计证据。
第一阶段不执行自动改价、批量发券、商品下架等高风险动作。验证重点放在“读取、对比、分析、确认、留痕”，审计和回滚能力成熟后再扩大自动执行范围。

9. 联调规范

启动会主文档只定义联调规范边界，具体字段结构放到 API 文档中维护。两队必须围绕以下对象保持一致。

规范对象 含义 由谁产生 由谁消费
connectToken 短期绑定令牌，用于把 AllMall 店铺授权记录与 MuleRun session 关联。 SaaS 团队 Agent 团队 / MuleRun
runtimeSessionId MuleRun 持久化浏览器 profile/session 的引用。 Agent 团队 / MuleRun SaaS 团队
taskId / runId ArkOps 任务与执行实例标识，用于串联时间线、事件和 artifact。 SaaS 团队 Agent 团队 / MuleRun
eventType Agent 执行事件，例如 run_started、step_completed、approval_required、login_required、run_failed。 Agent 团队 / MuleRun SaaS 团队
artifact 执行证据，包括截图、日志、trace、报表或模型摘要。 Agent 团队 / MuleRun SaaS 团队 / 审计模块
具体 JSON、错误码、签名机制、分页过滤、状态机枚举统一在 API 文档和数据库文档中维护。本文件只保留启动会需要理解的协作边界。

10. 里程碑与验收标准

M1
独立启动

目标

SaaS 团队完成 Portal 骨架和 mock API；Agent 团队完成 Login Bootstrap Agent 原型；双方冻结最小事件集。

验收

M2

验收

首个业务 Agent

目标

竞品调研 Agent 跑通任务输入、步骤执行、截图日志、数据来源和结果摘要。

目标

验收

11. 启动会解读重点

双团队总体说明

本阶段不是等待完整平台建设完成后再启动 Agent，而是由 SaaS 团队和 Agent 团队并行推进。SaaS 团队建设控制台、任务、审批和审计能力；Agent 团队在 MuleRun 中完成真实登录、真实执行和真实证据回传。两边通过最小规范联调，保持独立开发与持续集成。

内测阶段 MuleRun 承担主要执行实验能力，业务主控权归属 AllMall。MuleRun 是执行实验室，AllMall 是业务控制台。团队先证明业务闭环，再进入 runtime 替换、自研或并存的架构决策。

Agent 团队关键任务

  • 第一阶段目标是证明真实平台执行能力，不是开发 SaaS 后台。
  • 输出可复用 SOP、截图日志、结果摘要和失败场景清单，而不是一次性 demo。
  • 人机校验、session 失效、平台风控和页面变化必须沉淀为产品状态、事件和人工确认机制。

12. 每周协作机制

机制 参与者 输出
每周联调会 SaaS 负责人、Agent 负责人、产品/业务负责人 演示最新闭环、确认本周阻塞、更新问题清单。
接口规范同步 后端负责人、Agent 工程负责人 确认 eventType、payload、artifact、approval_required 是否有变更。

13. 后续演进路线

Beta

手动 MuleRun 联调

connectToken + 用户手动打开 MuleRun Agent。目标是最快跑通真实登录、真实执行和真实回调。

Productization

Runtime API 接入

如果 MuleRun 开放 session/run API，AllMall 后端可自动创建登录 URL 或启动任务。

Mature

多 Runtime 并存

通过 Agent Runtime Adapter 接入 MuleRun、自研 Claw Runner 和平台 API Runtime，保持控制面稳定。

不过早自研完整 runtime，也不把业务控制权锁死在 MuleRun。先用 MuleRun 跑出真实业务闭环，再根据可控性、稳定性、成本和客户部署要求做下一阶段选择。
