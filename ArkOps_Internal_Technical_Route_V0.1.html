ArkOps Internal Route · V0.1

内部技术路线与 MuleRun 联调方案

本文档用于记录 ArkOps 内测阶段的工程分工、MuleRun Agent 接入方式、店铺登录会话绑定流程，以及未来从 MuleRun 验证期演进到标准 Agent Runtime 的路线。

1. 阶段目标

内测阶段的核心目标不是一次性完成最终架构，而是让两个团队尽早并行研发，并通过一个足够窄、足够清晰的联调契约完成真实场景验证。

SaaS 平台团队

ArkOps Control Plane
  • 租户、用户、角色、店铺授权模型
  • 任务中心：Task / Workflow / Run / Step
  • 审批、审计、额度、计费和告警
  • connectToken 生成、校验和绑定接口
  • Agent 事件回调、日志和截图展示

Agent 团队

MuleRun Runtime Lab
  • 在 MuleRun Web 平台调试高价值 Agent
  • 验证 TikTok Shop 等平台的浏览器会话复用
  • 输出执行状态、截图、日志和结果摘要
  • 识别需要人工审批或重新登录的场景
  • 沉淀可复用的 Agent SOP 和输入输出格式
内测阶段允许 MuleRun 承载更多执行权限和浏览器会话能力，但 ArkOps 仍然需要保留租户、店铺、任务、审批、审计和状态流转的主控权。

2. 总体集成思路

当前先采用 connectToken + 用户手动打开 MuleRun Agent 的方式进行联调。ArkOps 不直接拉起 MuleRun 浏览器沙盒，而是生成一次性绑定令牌；用户在 MuleRun Web 平台手动启动指定 Agent，并把绑定令牌交给 Agent 完成回传绑定。 

ArkOps Portal
  -> 创建店铺授权记录
  -> 生成 connectToken
  -> 提示用户打开 MuleRun Login Bootstrap Agent

MuleRun Agent
  -> 用户输入 connectToken
  -> 打开浏览器沙盒 / profile
  -> 用户手动登录店铺并完成人机校验
  -> MuleRun 持久化 browser session
  -> 回调 ArkOps 完成 storeId 与 runtimeSessionId 绑定

这样做的好处是 SaaS 平台团队和 Agent 团队不会互相阻塞。即使 MuleRun 暂时没有开放完整的外部创建 session API，也可以先完成真实登录、真实执行和真实回调链路。

3. 店铺登录会话绑定流程

对 TikTok Shop 等平台，登录经常涉及 captcha、2FA 或其他人机校验。ArkOps 不应把这类动作设计为自动绕过，而应设计为 Human Session Bootstrap / Re-authentication。

  1. 用户在 ArkOps 中添加 TikTok Shop 店铺。
  2. ArkOps 创建 StoreAuthorization，状态为 pending_login。
  3. ArkOps 生成短期有效的 connectToken。
  4. 用户在 MuleRun Web 平台打开 TikTok Shop Login Bootstrap Agent。
  5. 用户将 connectToken 输入 MuleRun Agent。
  6. MuleRun 打开专属浏览器沙盒 / profile。
  7. 用户手动登录 TikTok Shop，并完成 captcha、2FA 或其他校验。
  8. MuleRun 持久化浏览器 profile、cookie 或 session。
  9. MuleRun Agent 调用 ArkOps 绑定接口，回传 runtimeSessionId。
  10. ArkOps 将店铺状态更新为 connected。
{
  "connectToken": "arkops_connect_xxx",
  "storeId": "store_001",
  "platform": "tiktok_shop",
  "runtimeProvider": "mulerun",
  "runtimeSessionId": "mr_session_xxx",
  "status": "connected"
}
内测阶段可以让 MuleRun 保存 browser session，但 ArkOps 不应把长期店铺密码、长期 cookie 或完整 token 当作普通任务参数反复传递。ArkOps 应保存 runtime session reference，并保留吊销、重绑和审计能力。

4. Agent 任务执行联调

店铺完成绑定后，ArkOps 创建具体业务任务，并把任务目标、约束条件和 MuleRun session reference 交给 MuleRun Agent。MuleRun 负责加载对应浏览器 profile，执行 Agent SOP，并把过程事件回传 ArkOps。 

{
  "taskId": "task_123",
  "runId": "run_456",
  "tenantId": "tenant_a",
  "storeId": "store_001",
  "platform": "tiktok_shop",
  "agentType": "ads_optimizer",
  "runtimeProvider": "mulerun",
  "runtimeSessionId": "mr_session_xxx",
  "goal": "分析低 ROI 广告计划并提出预算调整建议",
  "constraints": {
    "maxBudgetChangePercent": 15,
    "requiresApprovalAboveAmount": 500
  },
  "callbackUrl": "https://api.arkops.example.com/agent-events"
}

建议事件类型

  • run_started：Agent 已开始执行。
  • step_started：某个步骤开始。
  • step_completed：某个步骤完成，可附带截图、日志和摘要。
  • approval_required：需要人工审批，例如大额预算调整、批量改价或风险内容。
  • login_required：会话失效或再次出现人机校验，需要用户重新登录。
  • run_succeeded：任务完成。
  • run_failed：任务失败，需要记录失败原因和可回放证据。

5. 安全与边界

  • connectToken 必须短期有效：建议一次性使用，绑定成功后立即失效。
  • 绑定必须校验租户与店铺：ArkOps 需要确认 token 对应的 tenant、store 和 platform。
  • 敏感信息最小化传递：任务 payload 中只传 session reference，不传长期密码或完整 cookie。
  • 所有回调进入审计日志：包括绑定、登录失效、审批请求、执行步骤、截图和最终结果。
  • 保留吊销能力：租户可以在 ArkOps 中断开店铺，ArkOps 将 session 标记为 revoked，并通知 MuleRun 停止后续使用。

6. 里程碑

M1：独立研发

  • ArkOps 完成租户、店铺、任务和 connectToken 基础模型。
  • MuleRun 完成 Login Bootstrap Agent 原型。
  • 双方约定任务 payload 和事件回调格式。

M2：绑定联调

  • 用户通过 connectToken 完成 TikTok Shop 登录绑定。
  • ArkOps 能展示店铺 connected / login_required / revoked 状态。
  • MuleRun 能回传 runtimeSessionId。

M3：首个高价值 Agent

  • 优先选择广告监控或商品上架 Agent。
  • 跑通任务创建、执行、截图、日志、审批和结果回传。
  • 沉淀可复用 SOP 模板。

M4：演进与替换能力

  • 如果 MuleRun 提供 session API，升级为 ArkOps 后端自动创建登录 URL。
  • 抽象 Agent Runtime Adapter，避免平台绑定。
  • 评估自建 Claw Runner 或混合 runtime。

7. 未来演进路线

Beta 阶段：
connectToken + 用户手动打开 MuleRun Agent

产品化早期：
ArkOps Backend 调 MuleRun API 创建 session / run
ArkOps Frontend 打开 MuleRun 返回的登录 URL

成熟阶段：
ArkOps Agent Runtime Adapter
  -> MuleRun Runtime
  -> Self-hosted Claw Runner
  -> Platform API Runtime

长期目标：
ArkOps 保持控制面稳定，runtime 可以替换或并存。
