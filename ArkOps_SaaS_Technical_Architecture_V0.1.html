ArkOps SaaS Architecture · V0.1

SaaS 平台整体技术架构

本文档描述 ArkOps SaaS 平台第一阶段的整体工程架构，聚焦租户、店铺、权限、任务、审批、审计、额度、计费和 Agent Runtime Adapter 等控制面能力。

1. 架构定位

ArkOps SaaS 平台是整个系统的控制面，不直接追求在第一阶段完成最强 Agent 能力，而是先建立可治理、可审计、可扩展的业务平台底座。

  • 业务主控：租户、用户、角色、店铺、套餐和账单归 ArkOps 管理。
  • 执行编排：业务目标被抽象为 Task / Workflow / Run / Step。
  • 风险控制：高风险动作进入 Approval，所有执行进入 Audit Log。
  • Runtime 解耦：MuleRun 是内测阶段 runtime provider，后续可以替换或并存自研 Claw Runner。

2. 推荐系统分层

Frontend Portal
  -> API Gateway / BFF
  -> Auth & Tenant Context
  -> Domain Services
       - Tenant Service
       - User & RBAC Service
       - Store Authorization Service
       - Task Control Service
       - Approval Service
       - Audit Service
       - Billing & Quota Service
       - Notification Service
       - Agent Runtime Adapter
  -> Infrastructure
       - PostgreSQL
       - Redis
       - Queue / MQ
       - Object Storage
       - Secret Vault
       - Observability
第一阶段可以采用模块化单体或轻量服务化架构，优先保证领域边界清晰。是否拆微服务不应早于核心模型稳定。

3. 前端 Portal 模块

租户与用户

  • 公司 / 租户管理
  • 用户、角色、权限配置
  • 成员邀请、禁用和审计

店铺授权

  • 添加平台店铺
  • 生成 connectToken
  • 展示 connected / pending_login / login_required / revoked 状态

任务中心

  • 创建 Task
  • 查看 Run / Step 时间线
  • 查看截图、日志和执行摘要

审批与审计

  • 审批高风险动作
  • 查看审批理由和前后值
  • 按租户、店铺、Agent、任务检索审计日志

4. 确定技术栈

第一阶段技术栈在本文档中直接确定，后续研发按照该方案推进。目标不是追求技术复杂度，而是尽快交付稳定的 SaaS 控制面。

层级确定技术说明
前端框架React + TypeScript用于构建 ArkOps Portal 中后台应用，统一使用函数组件、Hooks 和类型化接口模型。
前端构建Vite适合中后台应用，开发启动快，工程配置简单。
UI 组件Ant Design用于表格、表单、权限管理、任务时间线、审批列表等后台场景，减少早期 UI 基建成本。
后端框架Node.js + NestJS + TypeScript统一前后端 TypeScript 技术体系，NestJS 负责模块化后端、依赖注入、权限守卫和 API 分层。
ORM / 数据访问Prisma负责 PostgreSQL schema migration、类型化查询和数据模型维护。
API 风格REST + Webhook/Event APIPortal 业务接口使用 REST；MuleRun / Agent Runtime 回调使用 Webhook/Event API。
主数据库PostgreSQL承载租户、店铺、任务、审批、审计、计费等核心业务数据。
缓存与锁Redis用于 connectToken TTL、分布式锁、任务节流、会话状态缓存。
队列BullMQ + Redis用于任务投递、审批后续执行、通知、artifact 后处理和异步审计归档。
对象存储S3-compatible Object Storage开发和内测环境使用 MinIO；生产环境使用兼容 S3 协议的云对象存储，统一保存截图、trace、报表和日志归档。
Secret VaultHashiCorp Vault用于保存店铺授权凭证、Webhook 密钥、代理配置、LLM Key 等敏感信息引用。
部署Docker Compose for beta, Kubernetes for production内测阶段使用 Docker Compose 快速部署；生产阶段迁移到 Kubernetes 管理服务、队列 worker 和扩展能力。
可观测OpenTelemetry + Prometheus + Grafana + Loki统一采集 API、Agent 回调、队列任务、错误日志和关键业务指标。
后续文档中不再保留多套候选技术栈。若需要更换某项技术，应通过版本化架构决策记录说明原因、影响范围和迁移计划。

5. 后端核心服务

Auth & Tenant Context

负责登录态、租户上下文、角色权限、数据访问边界。所有 API 必须显式解析 tenantId，并做权限校验。

Store Authorization Service

负责店铺授权状态、connectToken、MuleRun runtimeSessionId 绑定、吊销和重新登录流程。

Task Control Service

负责 Task / Workflow / Run / Step 的创建、状态流转、重试、暂停、失败和结果归档。

Approval Service

负责高风险动作审批，支持 Portal、飞书、钉钉、企业微信等通道扩展。

Agent Runtime Adapter

负责把 ArkOps 标准任务格式转换为 MuleRun 或其他 runtime 能执行的格式，并把 runtime 事件转换回 ArkOps 标准事件。

Artifact Service

负责截图、浏览器 trace、执行日志、模型摘要和报表文件的上传、签名访问、归档和生命周期管理。

Notification Service

负责站内通知、Webhook、飞书、钉钉、企业微信等消息通道，支撑审批、登录失效、任务失败和异常告警。

6. 后台与数据库体系

第一阶段建议将数据库体系分为四类，分别服务于核心事务、短期状态、异步调度和执行证据。

PostgreSQL：核心事务库

  • 租户、用户、角色、店铺授权
  • Task / Run / Step / Approval
  • Quota、Billing、Audit Log 元数据
  • 建议所有业务表显式携带 tenant_id

Redis：短期状态与协调

  • connectToken TTL 和一次性消费
  • 任务节流、分布式锁、幂等键
  • 短期运行状态和前端轮询缓存

Queue / MQ：异步任务

  • 任务投递到 Agent Runtime Adapter
  • 审批通过后的继续执行事件
  • 通知、账单统计、审计归档等后台任务

Object Storage：执行证据

  • 关键步骤截图
  • 浏览器 trace 和异常截图
  • Agent 输出报表、日志文件和模型摘要
数据库文档负责定义表结构细节；本架构文档只定义系统中每类存储承担的职责和边界。

7. 内测阶段与 MuleRun 的关系

内测阶段先采用 MuleRun Web 平台手动启动 Agent 的方式。ArkOps 通过 connectToken 完成店铺与 MuleRun session 的绑定。

ArkOps owns:
  tenant / store / task / approval / audit / quota / billing

MuleRun owns during beta:
  agent workflow experiments / browser sandbox / persisted browser profile

Integration contract:
  connectToken
  runtimeSessionId
  task payload
  agent event callback

8. 第一阶段交付边界

  • 完成租户、用户、角色、权限和店铺授权基础模型。
  • 完成 connectToken 绑定流程和店铺授权状态机。
  • 完成 Task / Run / Step / Approval / Audit Log 基础能力。
  • 完成 MuleRun Agent 回调事件接收和展示。
  • 完成基础 quota、billing 记录和操作额度控制。
  • 暂不要求自研完整 Worker Runtime，但需要预留 Agent Runtime Adapter 抽象。

9. 项目里程碑与时间预估

以下时间以一个小型全职 SaaS 团队为估算基础：前端 1-2 人、后端 2-3 人、测试/产品/设计按需参与。若团队并行度更高，可压缩日历时间；若同时承担 MuleRun Agent 调试，则需要单独计算 Agent 团队工作量。

阶段周期预估关键交付验收标准
M0：需求与技术定稿 1-2 周 架构文档、数据库文档、API 文档、MuleRun 联调契约 两个团队确认字段、状态机、接口和联调边界。
M1：SaaS 基础框架 2-3 周 登录、租户、用户、角色、权限、基础布局、API 框架、数据库迁移 用户可登录 Portal，并在租户边界内管理成员和角色。
M2：店铺授权与 connectToken 2 周 店铺管理、connectToken 生成、MuleRun session 绑定回调、状态机 能完成 pending_login -> connected -> login_required / revoked 的状态流转。
M3：任务控制面 3-4 周 Task / Run / Step、任务时间线、Agent event callback、artifact 展示 MuleRun Agent 能把执行事件、截图、日志回传并展示在 ArkOps。
M4：审批、审计、通知 2-3 周 Approval、Audit Log、站内通知、Webhook/飞书/钉钉预留 高风险动作可进入审批，审批结果和执行证据可审计。
M5：额度、计费与内测加固 2-3 周 Quota、Billing 基础记录、异常处理、权限补强、日志监控 支持少量内测租户稳定使用，并能追踪资源消耗和失败原因。
粗略估算：SaaS 平台 MVP 从文档冻结到可内测约 10-15 周。若先砍掉计费、复杂 RBAC 和多通知通道，只做单租户/少量内测租户，最小联调版本可压缩到 6-8 周。

10. 开发顺序建议

  1. 先冻结核心状态机：StoreAuthorization、Task、Run、Step、Approval。
  2. 先做可跑通的纵向闭环：创建店铺 -> connectToken -> MuleRun 绑定 -> 创建任务 -> 接收事件 -> 展示日志。
  3. 再扩展横向能力：更多角色权限、更多通知通道、更多平台店铺、更多 Agent 类型。
  4. 最后做生产化增强：Quota、Billing、审计归档、告警、部署弹性和安全加固。
不要一开始把所有后台页面都做满。优先做能支撑 MuleRun 联调和第一个高价值 Agent 的最短业务闭环。

11. 主要风险与前置验证

风险影响前置验证
MuleRun session 复用能力不确定影响店铺登录绑定和后续任务复用优先验证 browser profile 是否可持久化、可通过 runtimeSessionId 复用。
Agent 回调格式不稳定影响任务时间线和审计展示先定义最小事件集：run_started、step_completed、approval_required、login_required、run_failed。
多租户权限边界复杂可能造成数据越权从第一天要求所有业务表带 tenant_id，所有 API 统一注入 tenant context。
审批链路打断 Agent 执行影响自动化体验先支持 waiting_approval 状态和继续执行事件，复杂审批通道后置。
截图和 trace 存储增长快影响成本和查询效率制定 artifact 生命周期策略，内测期按租户和任务设置保留周期。
