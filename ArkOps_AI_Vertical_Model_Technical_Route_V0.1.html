ArkOps AI Vertical Model · V0.1

电商垂类模型研发技术架构与路线

本文档定义 ArkOps/AllMall 在电商自动化运营场景中的 AI 垂类模型落地方式。核心原则是：短期不从零训练基础大模型，而是通过通用大模型、RAG、工具调用、任务编排和审计闭环快速上线；中长期基于真实店铺数据、Agent 执行轨迹、人工审批反馈和经营结果，逐步形成可训练、可评测、可部署的电商运营垂类模型能力。

1. 模型定位

不是基础大模型

ArkOps 早期不承担从零预训练 LLM 的成本和风险。系统优先接入成熟通用模型 API，并把研发重点放在电商场景数据、任务流程、工具调用、风险控制和评测体系上。

降低研发风险 减少 GPU 重资产

是电商运营模型能力

垂类模型能力由四类资产组成：电商知识、任务规划能力、Agent 行为数据、经营结果反馈。长期目标是让模型更懂店铺上下文、平台规则和真实业务指标。

任务数据 执行轨迹 业务反馈
BP 表述建议：AllMall 的技术壁垒不是单纯拥有一个通用模型，而是通过真实电商运营任务持续沉淀“业务上下文 + Agent 执行轨迹 + 人工审批反馈 + 经营结果”的专有数据集，并基于该数据集构建电商运营垂类模型。

2. 总体技术架构

1

数据源层

来自店铺、平台规则、Agent 运行过程和业务结果的原始数据。

店铺业务数据商品、订单、库存、广告、评价、客服、退款、销售报表。
平台规则数据Amazon、TikTok Shop、Shopify、Shopee 等规则与运营 SOP。
Agent 执行数据Task、Run、Step、工具调用、浏览器动作、截图、日志、失败重试。
经营反馈数据CTR、CVR、ACOS、ROI、退款率、库存周转、客服响应时长。
2

数据治理与标注层

把自然产生的业务流程转成结构化训练样本和评测样本。

任务标签任务类型、意图、目标指标、所需数据、风险等级。
过程标签计划质量、步骤是否成功、工具调用结果、错误类型。
审批标签通过、修改后通过、驳回、需要补充信息、人工修正原因。
结果标签业务指标变化、客户采纳情况、任务成功率、复盘结论。
3

知识与检索层

早期优先使用 RAG，把平台规则、运营 SOP 和店铺上下文提供给通用模型。

知识库平台政策、广告策略、Listing 规范、客服话术、内部 SOP。
向量检索初期使用 PostgreSQL + pgvector，规模扩大后评估 Qdrant/Milvus。
Reranker对召回内容重排，减少错误规则进入模型上下文。
上下文组装按租户、店铺、平台、任务类型动态构造 Prompt。
4

模型服务层

从通用模型 API 起步，逐步增加小模型、微调模型和模型路由。

通用 LLM任务理解、计划生成、复杂推理、运营建议。
小模型服务意图分类、风险识别、字段抽取、异常检测。
垂类微调模型SFT、LoRA、DPO，用于任务规划、客服、Listing、广告诊断。
模型路由复杂任务用强模型，高频简单任务用低成本模型。
5

Agent 执行与闭环层

模型输出不是单纯文本，而是计划、工具调用、审批请求和可审计执行记录。

Task Control PlaneTask、Run、Step、Queue、状态机、失败重试。
Runtime AdapterMuleRun Beta、平台 API Runtime、未来 Claw Runner。
HITL Approval高风险动作必须人工确认，审批结果反哺模型。
Audit & Artifact截图、日志、输入输出、差异对比、回滚线索。

3. 数据与标注规范

标注规范的目标不是建立重型标注工厂，而是把标注嵌入 Agent 执行与人工审批流程。每一次任务都自动产生基础样本，运营人员在审批、修改、驳回时补充少量结构化原因，系统再结合业务结果形成可训练数据。

数据类别 核心字段 用途 采集方式
任务输入数据 任务目标、店铺、平台、类目、时间范围、相关业务指标 训练意图识别、任务规划、上下文选择 用户创建任务、系统自动补全店铺上下文
Agent 执行轨迹 计划步骤、工具调用、页面动作、API 请求、截图、日志、错误码 训练 Agent Planner、Critic、失败恢复策略 Task/Run/Step 自动记录
人工审批反馈 通过、修改后通过、驳回、补充信息、修正原因、风险原因 训练风险判断、偏好优化、DPO 数据 HITL 审批表单结构化采集
经营结果反馈 CTR、CVR、ACOS、ROI、销售额、退款率、库存周转、客服响应 评估建议是否真正改善业务，而不是只判断文本是否正确 任务完成后按窗口期自动回收指标
规则与 SOP 数据 平台规则、广告策略、Listing 标准、客服模板、内部操作规范 RAG、规则校验、模型回答依据 人工维护 + 文档解析 + 版本管理

4. 模型能力拆分

电商意图识别模型

识别任务属于广告、Listing、客服、库存、订单、评价、报表等哪类场景，并提取目标指标和约束条件。

分类 抽取

风险门控模型

判断任务或步骤是否涉及改价、预算、退款、下架、批量修改等高风险动作，决定是否进入人工审批。

安全 HITL

运营任务规划模型

把经营目标拆成可执行步骤，选择数据源、工具、运行时和检查项，输出可审计的计划。

Planner Tool Calling

Listing 优化模型

围绕标题、五点描述、关键词、图片检查和平台规则，生成可解释的优化建议和改写版本。

内容生成 规则校验

广告诊断模型

分析广告报表、销售数据和库存状态，识别 ACOS 异常、预算浪费、关键词问题和投放机会。

诊断 建议

Agent Critic 模型

检查 Agent 的计划和执行结果是否遗漏关键步骤、违反规则或缺少证据，作为执行前后的质量控制器。

评审 纠错

5. 研发流程

1. 任务上线 选择高频、ROI 明确、风险可控的运营任务，例如广告巡检、Listing 检查、客服草稿。
2. 数据沉淀 Task/Run/Step 自动记录执行过程，审批流采集人工判断，任务完成后回收经营指标。
3. 评测集构建 从真实任务中筛选 gold/silver 样本，建立场景化评测集和回归测试集。
4. 模型迭代 先优化 Prompt/RAG，再做分类小模型，最后基于 SFT、LoRA、DPO 做垂类微调。
5. 灰度发布 使用 A/B、租户灰度、人工审批兜底和审计回放，持续观察任务成功率与业务指标。

6. 技术栈建议

模块 早期选择 中后期演进
通用模型 OpenAI、Claude、Gemini、DeepSeek、通义千问等 API，按任务成本和稳定性路由 保留强模型兜底，引入自部署小模型处理高频任务
RAG 与向量库 PostgreSQL + pgvector，结合对象存储保存原文和版本 数据规模扩大后评估 Qdrant、Milvus 或托管向量数据库
任务数据 沿用 ArkOps SaaS 数据模型：Task、Run、Step、Approval、Audit、Artifact、Outcome 增加数据集版本、样本质量分、评测集、训练集、模型版本表
数据处理 NestJS 后台任务 + BullMQ 队列，定期生成样本与评测数据 引入 Airflow/Dagster、ClickHouse 或数据湖用于大规模分析
训练方式 Prompt、RAG、Few-shot、小模型分类，不做重训练 SFT、LoRA、DPO、蒸馏、量化、模型路由
部署与观测 模型调用日志、Token 成本、延迟、失败率、人工修改率 模型网关、在线评测、漂移监控、自动回滚、GPU 推理优化

7. GPU 与人员需求

GPU 资源路线

  • MVP 阶段：基本不需要自有 GPU，使用通用模型 API 和云端 embedding/reranker 服务。
  • 数据沉淀阶段：需要少量云 GPU 做小模型实验、LoRA 微调、离线评测，可按需租用。
  • 垂类模型阶段：基于 7B/14B/32B 开源模型做 SFT、LoRA、DPO，仍以云 GPU 为主。
  • 规模化阶段：关注推理成本，采用量化、缓存、批处理、模型路由和部分自部署推理。

技术人员能力要求

  • 后端/平台：多租户数据建模、任务状态机、审计日志、权限与数据隔离。
  • Agent/LLM 工程：Prompt、RAG、Tool Calling、上下文管理、失败恢复。
  • 数据/算法：样本构造、评测集、embedding、分类、SFT、LoRA、DPO、模型部署。
  • 业务专家：广告、Listing、库存、订单、客服 SOP 和结果复盘规则。
早期最重要的不是招聘大模型研究员，而是把“可训练的数据闭环”做进产品。没有任务日志、审批反馈和经营结果，即使后期有 GPU，也很难形成真正的垂类模型壁垒。

8. 阶段路线图

0-3 个月

RAG + Prompt + 任务日志基础版

完成平台规则知识库、任务 Prompt 模板、Task/Run/Step 执行日志、Approval 审批字段和 Artifact 存储。模型以通用 LLM API 为主，不进行训练。

3-6 个月

标注规范与评测集 V0.1

围绕广告巡检、Listing 检查、客服草稿等首批场景，形成任务类型、风险等级、错误类型、人工修正原因、业务结果的结构化标注规范，沉淀第一批 gold/silver 样本。

6-12 个月

小模型与模型路由

训练或微调意图分类、风险门控、字段抽取、异常识别等小模型，降低通用模型调用成本。建立离线评测与线上灰度机制。

12-18 个月

电商垂类模型 v1

基于真实任务数据进行 SFT/LoRA 微调，重点覆盖广告诊断、Listing 优化、客服回复和 Agent Critic。引入 DPO 使用人工审批偏好数据优化模型输出。

18-36 个月

垂类模型 v2/v3 与自动化闭环

把模型能力与 Runtime Adapter、Claw Runner、平台 API Runtime 深度结合，形成任务规划、执行校验、风险控制、经营复盘的闭环。通过模型路由和推理优化持续降低单位任务成本。

9. 关键评测指标

模型质量

  • 任务意图识别准确率
  • 计划步骤完整率
  • 规则引用准确率
  • 幻觉率与无依据建议比例

执行质量

  • Agent 任务成功率
  • 工具调用失败率
  • 人工修改率
  • 风险动作拦截率

商业结果

  • 建议采纳率
  • ACOS/ROI 改善
  • 客服响应效率提升
  • 单任务 Token 成本下降

10. 主要风险与控制

风险 表现 控制方式
数据质量不足 样本字段缺失、人工反馈过粗、经营结果无法回收 把标注嵌入审批流程，设置必填修正原因，按任务窗口自动回收指标
模型幻觉 编造平台规则、给出无依据建议、忽略店铺约束 RAG 引用、规则版本管理、Agent Critic、强制输出证据链
自动执行风险 错误改价、错误预算调整、误删或误下架商品 风险门控模型 + HITL 审批 + Run/Step 审计 + 回滚线索
训练成本失控 过早自研大模型、GPU 使用率低、模型维护复杂 先 API 和 RAG，后小模型，再 LoRA/DPO；云 GPU 按需租用
多租户数据合规 不同租户数据混用、敏感信息泄露、训练数据不可追溯 租户隔离、PII 脱敏、样本血缘、授权范围控制、审计日志
